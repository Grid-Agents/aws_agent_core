Running terraform apply reads the recipe and builds a tiny rented computer in Amazon’s data centre that sits between your web app and the heavy AI service — taking requests, doing small jobs (like reading emailed PDFs), and forwarding the big ones.

What even is this? (30-second primer)

i

Terraform = “infrastructure as a recipe”

why these files exist
concept▶

Normally you’d build a server by clicking around Amazon’s website. That’s error-prone and nobody remembers what they clicked. Terraform lets you instead describe the server in text files. You run one command and it builds exactly that — and it’s reviewable, repeatable, and version-controlled like code. This is called Infrastructure as Code.

Two building blocks you’ll see everywhere below: a resource = “create this thing”, and a data source = “look up something that already exists” (read-only).

main.tf — walking the file top to bottom

user_data.sh.tftpl — what happens when the box turns on

▸

The first-boot script

runs once, as root, the moment the server starts
compute▶

An EC2 user-data script. Terraform fills in the blanks (region, model, the Gmail settings) and AWS runs it on first boot:

  1. Install basics (tar, unzip) and the AWS CLI.
  2. Install uv (a fast Python package/version manager).
  3. Download the app’s code zip from S3 (using the box’s badge — no GitHub keys needed) and unpack it.
  4. Write /opt/grid-bff/bff.env — the settings the app reads (region, model, AgentCore ARN, and the Gmail intake switches). No secrets here — the token is read from AWS at runtime.
  5. Install the app’s Python dependencies (uv sync --extra build).
  6. Create a systemd service so the app starts now and on every reboot, auto-restarting if it crashes.
Because it’s one service (no --workers), exactly one email poller runs — no risk of two copies grabbing the same email.

variables.tf & outputs.tf — the knobs and the receipts

⚙

variables.tf — the inputs you pass in

terraform apply -var "..."
config▶

Knobs so the same recipe works in any account: region, instance_type (t3.medium by default), runtime_arn (which AI brain to forward to), the S3 bucket, and the email-intake toggles (gmail_intake_enabled, gmail_token_ssm_param, gmail_query — all off by default).

📤

outputs.tf — what you get back after building

terraform output
config▶

After apply, Terraform hands you the instance_id and — most usefully — a ready-made SSM tunnel command. Run it and the private box’s console opens at http://localhost:8000 on your laptop — no open ports, no public IP needed.

Glossary

The lifecycle in one line: edit a .tf file → terraform plan (preview the diff) → terraform apply (AWS builds/changes it). Changing the boot script replaces the box entirely (that’s the user_data_replace_on_change flag — which is why on-disk data doesn’t survive a redeploy).
